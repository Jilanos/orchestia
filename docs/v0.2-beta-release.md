# Orchestia v0.2-beta Release

## Release

- Release name: `v0.2-beta`
- Release date: 2026-05-26
- Reviewed Orchestia commit: `28deaf4 Add v0.2-beta cockpit action layer`
- Release decision: accept

Orchestia `v0.2-beta` is not production-ready. It remains a human-supervised local orchestration system with explicit controls for Codex execution, review decisions, Loop state advancement, push, merge, and release tagging.

## Summary

`v0.2-beta` consolidates the first completed real mini-project loop and the first safe cockpit action layer:

- `job-offer-analyzer` completed the local MVP path from initial need through validation/docs.
- `job-offer-analyzer` was published to GitHub on the `integration` branch through controlled publication.
- The local cockpit now supports draft need intake, Loop dashboard, iteration timeline, token evidence scanning, and safe autonomous-loop instruction/stop actions.
- Browser actions remain limited to draft/control files under `task-runs/`.
- Codex execution, controlled Git flow execute, push, merge, and Logics promotion remain outside the cockpit action layer.

## What Is Supported

- State-driven Loop records and Logics memory for needs, requests, backlog items, tasks, and reviews.
- Explicit Codex execution through `codex exec --sandbox workspace-write` from verified workspaces.
- Autonomous local loop cycles with evidence capture and guarded auto-accept.
- Human-reviewed Logics decisions: `accept`, `revise`, `split`, or `reject`.
- Controlled Git flow dry-run and execute paths under explicit CLI authorization.
- Local cockpit inspection for Loop state, task-runs, autonomous-loop evidence, reviews, debug status, iterations, and tokens.
- Draft-only need intake from the cockpit.
- Safe cockpit POST actions for autonomous-loop instruction and stop files.

## What Is Still Manual

- Promoting cockpit need drafts into Logics records.
- Choosing the next primary need when ambiguous.
- Declaring firm blockers.
- Authorizing Codex execution.
- Reviewing evidence before acceptance.
- Authorizing push or merge through CLI guardrails.
- Creating release commits and tags.
- Operating any published project beyond its controlled branch policy.

## Known Limitations

- The cockpit has no authentication layer and is intended for localhost operation.
- Cockpit need intake creates drafts only; reviewed Logics promotion remains manual.
- Token usage parsing is best-effort and evidence-based only.
- The iteration timeline is inferred from filenames and simple Markdown parsing.
- Autonomous-loop advancement still needs explicit next-state data.
- Project-specific allowed-file enforcement remains prompt-and-review based.

## Risks

- Browser POST actions increase write surface and must remain limited to `task-runs/`.
- `workspace-write` Codex execution remains powerful and must stay behind explicit CLI authorization.
- Controlled Git flow execute paths can publish branches and must remain guarded.
- Published `job-offer-analyzer` is a small local CLI MVP, not a production service.

## Non-Goals

- Full autonomy.
- Background agents or daemons.
- Browser-triggered Codex execution.
- Browser-triggered push, merge, or controlled Git flow execute.
- Secret management automation.
- External service or billing API integration.
- Production hosting.
- Force push, branch deletion, rebase, or automatic tag creation.
