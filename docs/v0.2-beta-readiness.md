# v0.2-beta Readiness

## Current State Summary

Orchestia is at the post-`v0.2-alpha` state with the real `job-offer-analyzer` mini project completed and published. The latest relevant Orchestia commit before this readiness task is `e6ac45c Complete job offer analyzer publication`.

## v0.2-alpha Capabilities

- Loop state files can point to prepared Codex prompts.
- Controlled auto-loop and autonomous-loop runs capture evidence under `task-runs/`.
- Codex execution is explicit and uses `codex exec --sandbox workspace-write`.
- Controlled Git flow can dry-run and execute guarded branch pushes and merges.
- The local cockpit can inspect Loop states, `task-runs/`, Logics records, reviews, auto-loop evidence, autonomous-loop evidence, and debug status.

## Job Offer Analyzer Validation Summary

- `IN-0002` is complete.
- `PN-0005`, `PN-0006`, `PN-0007`, and `PN-0008` are complete.
- `LS-0002` is in completed state.
- Local project commit `96efb67 Add validation and documentation` was accepted.
- The project was published to `https://github.com/Jilanos/job-offer-analyzer` on branch `integration`.
- `main` and `master` were not targeted.

## Current Cockpit Limitations

- It cannot intake a new user need from the browser.
- It cannot create safe draft records.
- It does not provide a compact cross-run iteration timeline.
- It does not provide a token usage evidence dashboard.
- Autonomous-loop instructions and stop requests are copyable CLI commands rather than browser actions.
- It does not provide an explicit local action layer.

## v0.2-beta Target Scope

- Add draft-only need intake under `task-runs/`.
- Add a compact Loop dashboard.
- Add an iteration timeline across runs, reviews, and tasks.
- Add token usage evidence parsing when token data is present.
- Add safe browser POST actions for autonomous-loop instruction and stop files.
- Keep push, merge, Codex execution, and arbitrary command execution out of the cockpit.

## Readiness Decision

Decision: accept.

The prerequisites are present: Orchestia is clean, latest project evidence is recorded, `job-offer-analyzer` completion and publication records exist, and the cockpit action layer can proceed safely within the authorized scope.

## Risks

- The action layer adds write paths and must stay limited to `task-runs/`.
- Token usage parsing can only report evidence that appears in local text files.
- Browser actions must not become a general command runner.
- The cockpit is still localhost-only and has no authentication layer.

## Implementation Plan

1. Add need intake draft pages and POST handling.
2. Add Loop dashboard and iteration timeline pages.
3. Add token usage evidence scanning and display.
4. Add safe autonomous-loop instruction and stop POST actions.
5. Add read-only autonomous-loop status support to `scripts/orchestia_loop.sh`.
6. Update cockpit, workflow, and roadmap documentation.
7. Validate with syntax checks, cockpit smoke tests, and disposable POST smoke tests.

## Non-Goals

- No Codex execution from the browser.
- No push or merge from the browser.
- No arbitrary command execution.
- No dependency installation.
- No external services or billing APIs.
- No secret inspection.
- No release tag.

## Validation Plan

- `bash -n scripts/orchestia_loop.sh`
- `python3 -m py_compile scripts/orchestia_ui.py`
- Local cockpit GET smoke tests for dashboard, needs, loop dashboard, iterations, tokens, autonomous-loop, and debug pages.
- Disposable need-intake POST smoke test under `task-runs/`.
- Autonomous-loop instruction and stop POST smoke test against an existing `task-runs/*-autonomous-loop` directory when available.
- Documentation grep checks for beta readiness, action layer, token usage, and review decisions.
