# v0.3 Roadmap: Cockpit-Driven Orchestration

## Current Baseline

The baseline is Orchestia `v0.2-beta`.

`v0.2-beta` provides:

- state-driven Logics records;
- explicit Codex execution through CLI-controlled loops;
- autonomous-loop evidence capture;
- controlled Git flow guardrails;
- completed and published `job-offer-analyzer` MVP evidence;
- local cockpit pages for inspection, draft need intake, iteration timeline, token evidence, and safe autonomous-loop instruction/stop files.

## Problem Statement

The cockpit can prepare and monitor orchestration, but it cannot yet drive the core workflow end to end. Need intake drafts remain disconnected from Logics draft records, task prompt preparation is still manual, autonomous-loop launch still happens only from the terminal, and review finalization still requires CLI or direct file edits.

## Target Outcome

Orchestia `v0.3` should make the cockpit the primary local operator surface for driving orchestration while preserving explicit human confirmations and CLI-grade safety boundaries.

## Supported User Journey

1. User opens the local cockpit.
2. User enters a new need.
3. Cockpit creates a need intake draft.
4. Cockpit helps generate Logics draft records.
5. Human reviews and accepts Logics drafts.
6. Cockpit prepares a task prompt draft.
7. Human confirms autonomous-loop execution.
8. Cockpit shows iterations, evidence, token usage if available, and blockers.
9. Human can add instructions or stop the loop.
10. Work advances until completion or blocker.
11. Publication remains controlled and explicit.

## Scope

- Cockpit-driven draft generation for Logics records.
- Prompt draft preparation from accepted task scope.
- Explicitly confirmed autonomous-loop launch from the cockpit.
- Refreshed or live-enough loop status display.
- Token usage evidence improvements.
- UI support for explicit review decisions.
- Controlled Git flow handoff preparation from the cockpit.
- Negative-path validation for cockpit actions.

## Non-Goals

- No hidden background agents.
- No browser action without explicit human confirmation.
- No arbitrary command execution.
- No direct browser push or merge without controlled Git flow guardrails.
- No direct mutation of final Logics records without review.
- No external service integration.
- No secret or credential access.
- No production hosting.

## Safety Model

- The cockpit remains local-first and localhost-oriented.
- Every write action has a named destination and bounded schema.
- Draft actions write under `task-runs/` until reviewed.
- Final Logics changes require explicit review and acceptance.
- Codex execution requires explicit confirmation and an allowed command path.
- Push and merge remain behind controlled Git flow.
- Ambiguous next primary needs and firm blockers remain human-owned decisions.

## Cockpit Actions Roadmap

1. Convert need intake into Logics draft records.
2. Promote reviewed Logics drafts into final records.
3. Prepare task prompt drafts from accepted tasks.
4. Launch autonomous-loop with explicit confirmation.
5. Append instructions and stop requests to active runs.
6. Finalize reviews from UI with explicit decision.
7. Prepare controlled Git flow handoff commands.
8. Record action evidence under `task-runs/`.

## Execution Model

- Dry-run preview is shown before any execution.
- Autonomous-loop launch requires explicit user confirmation.
- The cockpit must show workspace, Loop state, prompt path, test command, and execute flags before launch.
- Executions write evidence under `task-runs/`.
- The UI must never infer accept/revise/split/reject without explicit policy or human input.

## Token Usage Model

- Token usage remains evidence-based.
- The cockpit parses local evidence when available.
- Missing token values are shown as unavailable, not estimated.
- Future Codex runs should emit machine-readable token evidence when possible.

## Review And Decision Model

- Reviews expose required evidence, findings, risks, and decision.
- UI decision controls must allow only `accept`, `revise`, `split`, or `reject`.
- Review finalization requires explicit reviewed task and review ID.
- Loop advancement requires accepted evidence and explicit next-state data.

## Publication Model

- Publication remains a separate controlled workflow.
- The cockpit can prepare controlled Git flow handoff and dry-run previews.
- Execute paths for push and merge require explicit confirmation and existing guardrails.
- `main` and `master` remain protected by default.

## Validation Strategy

- Unit-level syntax checks for scripts.
- Cockpit GET smoke tests.
- POST smoke tests against disposable `task-runs/` resources.
- Negative-path tests for missing confirmation, invalid decisions, path traversal, dirty workspaces, missing prompts, failed tests, and protected branch attempts.
- Disposable workspace validation before real project validation.

## Proposed Tasks

- Cockpit action: convert need intake into Logics draft.
- Cockpit action: prepare task prompt draft.
- Cockpit action: launch autonomous-loop with explicit confirmation.
- Cockpit view: live or refreshed loop run status.
- Cockpit view: token usage improvements.
- Cockpit action: finalize reviews from UI with explicit decision.
- Cockpit action: prepare controlled Git flow handoff.
- v0.3 negative-path validation suite.
- v0.3-alpha release readiness.

## Completion Criteria

- A new need can move from cockpit intake to reviewed Logics drafts.
- A task prompt can be prepared from cockpit-visible state.
- An autonomous-loop can be launched from the cockpit only after explicit confirmation.
- Evidence, blocker, token, and decision states are visible.
- Review finalization and Loop advancement remain explicit and validated.
- Push and merge remain controlled and guarded.
- Negative-path validation passes.

## Risks

- Browser-triggered execution increases operator risk.
- Draft-to-final promotion can blur review boundaries if not designed carefully.
- Token evidence may stay incomplete until runners emit structured usage.
- Live status can become misleading if it is not clearly refreshed or timestamped.
- More UI actions increase the need for path and action validation.

## Next Recommended Task

Design the cockpit need-intake-to-Logics-draft flow with disposable validation and no final Logics mutation from browser submission.
